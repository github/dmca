**What technological measures do you have in place and how do they effectively control access to your copyrighted material?**   
   
Ubisoft Entertainment SA applies four distinct technological protection measures to the videogame Rocksmith® 2014 Edition - Remastered - Learn & Play (Steam AppID 221680). Each uses a different AES key, embedded only within the Rocksmith 2014 client executable. None of the keys are published by Ubisoft, and none can be derived without reverse-engineering the client.   
   
TPM 1. PSARC archive encryption. Protects the outer container of all song content (audio, note charts, manifests, album art, gear artwork). AES-256, CFB mode, 128-bit segment size. The 32-byte key and the 16-byte initialization vector are embedded in the client.   
   
TPM 2. SNG note-chart encryption (PC build). Protects the binary note-chart files (.sng) that contain the per-instrument arrangements (Lead, Rhythm, Bass, Vocals) inside each PSARC. AES-256, CTR mode (implemented over ECB blocks), 16-byte IV read from the SNG header. The PC build of Rocksmith 2014 uses a 32-byte SNG key distinct from the PSARC key.   
   
TPM 3. SNG note-chart encryption (Mac build). Identical scheme as TPM 2, but the Mac build of Rocksmith 2014 uses a different 32-byte SNG key. The platform-specific keying means a single circumvention tool must reproduce both keys to interoperate across the PC and Mac builds of Rocksmith 2014.   
   
TPM 4. EVAS profile encryption. Protects player progression and save data. AES-256, ECB mode, no IV. The header of every Rocksmith 2014 profile file begins with the four-byte magic EVAS, which is how the circumvention tool identifies a profile to decrypt. The 32-byte key is again distinct from the three keys above.   
   
All four measures effectively control access to the protected works within the meaning of 17 U.S.C. § 1201(a)(3)(B). In the ordinary course of operation, accessing the protected content requires the application of information, namely the encryption key, and a process, namely the AES decryption routine, with the authority of Ubisoft through the Rocksmith 2014 client.

4.1 PSARC encryption (TPM 1)   
   
The file lib/psarc.py of github.com/byrongamatos/slopsmith, at lines 24 through 27, hardcodes the 32-byte Ubisoft AES key (hex literal [private]) and the 16-byte initialization vector (hex literal [private]). The function _decrypt_toc() on lines 30 through 32 instantiates AES.new(ARC_KEY, AES.MODE_CFB, iv=ARC_IV, segment_size=128) and applies it to decrypt the table of contents of any PSARC archive whose archive_flags field equals 4, which is the flag Ubisoft uses to denote encrypted Rocksmith 2014 archives. The docstring on line 1 of the file states verbatim "PSARC file extractor for Rocksmith 2014."   
   
The same key and initialization vector are reproduced in the file extract_assets.py of the slopsmith-plugin-tones repository. The script states verbatim that it is intended to "extract gear images from Rocksmith 2014's gears.psarc."   
   
The README of github.com/byrongamatos/slopsmith describes the implementation, at line 444, as a "Custom AES-CFB-128 decryptor with in-memory reading."   
   
The same PSARC key is also hardcoded in src/Rocksmith2014.PSARC/Cryptography.fs of github.com/byrongamatos/Rocksmith2014.NET, at the F# literal psarcKey (line 11). The desktop application at github.com/byrongamatos/slopsmith-desktop builds that file into the rscli binary that ships in every desktop release.   
   
4.2 SNG note-chart encryption, PC build (TPM 2)   
   
The file src/Rocksmith2014.SNG/Cryptography.fs of github.com/byrongamatos/Rocksmith2014.NET, at line 14, hardcodes the 32-byte Ubisoft SNG PC key (hex literal [private]). The function decryptSNG reads a 16-byte IV from the SNG header and applies the key via aesCtrTransformSIMD (SSE2-accelerated AES-CTR) to decrypt the chart. The compiled output ships in every github.com/byrongamatos/slopsmith-desktop release as the rscli binary, which is invoked from the github.com/byrongamatos/slopsmith pipeline to convert encrypted SNG files into XML for the highway renderer.   
   
4.3 SNG note-chart encryption, Mac build (TPM 3)   
   
The same file src/Rocksmith2014.SNG/Cryptography.fs of github.com/byrongamatos/Rocksmith2014.NET, at line 17, hardcodes a second 32-byte Ubisoft SNG key for the Mac build (hex literal [private]). The function getKey switches between the PC and Mac keys based on a Platform enum at runtime, so the rscli binary is able to decrypt SNG files from either platform's Rocksmith 2014 installation.   
   
4.4 EVAS profile encryption (TPM 4)   
   
The file routes.py of slopsmith-plugin-profileimport hardcodes the 32-byte EVAS profile key (the byte array starting 0x72, 0x8B, 0x36, 0x9E, 0x24, 0xED, 0x01, 0x34, ...). The accompanying comment identifies the key as the "EVAS profile decryption key (well-known in the Rocksmith community)." The module docstring states that the plugin "imports Rocksmith 2014 player stats from encrypted profile files."   
   
The same EVAS key is also hardcoded in src/Rocksmith2014.Common/Profile/Profile.fs of github.com/byrongamatos/Rocksmith2014.NET, at the F# literal profileKey (line 21). The file readHeader function verifies the four-byte magic EVAS before decrypting, so the code is specifically designed to operate against the Rocksmith 2014 profile format and no other.   
   
The compiled output of github.com/byrongamatos/Rocksmith2014.NET ships in every github.com/byrongamatos/slopsmith-desktop release. Together with the in-source Python implementation in slopsmith-plugin-profileimport, the project carries a redundant pair of EVAS implementations.
